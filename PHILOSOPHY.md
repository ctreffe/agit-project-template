# PHILOSOPHY.md

# Project Philosophy

The AGIT Project Template is based on the idea that good project work is easier to continue when intent, context, decisions and outputs are visible in the repository.

The repository should become the project memory.

---

# Intent Before Structure

A project should not start with a template-shaped answer.

It should start with the maintainer's intent:

- why the project exists
- what it should enable
- who it serves
- what should remain out of scope
- what a successful end state looks like

Structure should serve that intent.

---

# Context Over Chat History

Chat history is useful during collaboration, but it is not a reliable long-term project record.

Important context should be moved into repository documents, especially `PROJECT_CONTEXT.md`, PDRs and relevant documentation.

---

# Decisions Should Be Traceable

Important decisions should remain understandable after the conversation that produced them has ended.

Project Decision Records make durable decisions explicit without turning every discussion into process overhead.

---

# Lightweight by Default

The template should remain useful without becoming heavy.

Projects may remove, adapt or ignore parts that do not fit their actual work.

The goal is not to create bureaucracy. The goal is to make future continuation easier.

---

# General Projects May Become Dev Projects

Some projects begin as planning, research or documentation work and later become development-oriented.

The generic template should support that evolution. When the project becomes centered on code, automation, tests, releases or technical architecture, adopting the AGIT Dev Template may become appropriate.

---

# Integrity Over Appearance

Do not make work appear more complete than it is.

Incomplete work, uncertain assumptions, missing validation and open questions should be stated clearly.

Reliable collaboration depends on truthful project state.
